# Dockerfile Best Practices

> Rules for production-grade Dockerfiles. Apply these to every Dockerfile generated by this skill.

---

## Rule 1: Multi-stage builds for compiled languages

```dockerfile
# WRONG — Rust/Go/Java/.NET build deps + tools end up in final image
FROM rust:1.84
WORKDIR /app
COPY . .
RUN cargo build --release
CMD ["./target/release/myapp"]
# → ~2 GB image

# RIGHT — separate build stage, lean runtime
FROM rust:1.84 AS builder
WORKDIR /app
COPY Cargo.toml Cargo.lock ./
RUN mkdir src && echo "fn main() {}" > src/main.rs && cargo build --release  # cache deps layer
COPY src ./src
RUN cargo build --release

FROM debian:12-slim AS runtime
RUN apt-get update && apt-get install -y --no-install-recommends ca-certificates \
    && rm -rf /var/lib/apt/lists/*
WORKDIR /app
COPY --from=builder /app/target/release/myapp /app/myapp
USER 1000:1000
CMD ["/app/myapp"]
# → ~80 MB image
```

For Node, multi-stage is less dramatic but still useful when you have build deps (TypeScript compiler, sass, webpack):

```dockerfile
FROM node:22-alpine AS builder
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npm run build && npm prune --production

FROM node:22-alpine AS runtime
WORKDIR /app
COPY --from=builder --chown=node:node /app/node_modules ./node_modules
COPY --from=builder --chown=node:node /app/dist ./dist
COPY --from=builder --chown=node:node /app/package.json ./
USER node
EXPOSE 3000
CMD ["node", "dist/server.js"]
```

## Rule 2: Non-root user

```dockerfile
# Add a user (option A: use built-in 'node', 'nginx', etc. if image has one)
USER node

# Option B: create one explicitly
RUN addgroup -g 1000 -S app && adduser -u 1000 -S app -G app
USER app

# Option C: numeric only (works on any base, no /etc/passwd modification needed)
USER 1000:1000
```

Why: container escape vulnerabilities + general hardening. Root inside container can mean root on host in worst case (with --privileged or shared volumes).

When you MUST be root (rare): privileged port binding (<1024), system management. Better alternative: use port 8080 in container, map host's 80 → 8080.

## Rule 3: Pin base image versions exactly

```dockerfile
# WRONG
FROM node:latest
FROM node:22-alpine
FROM nginx

# RIGHT
FROM node:22.10.0-alpine3.21
FROM nginx:1.27.4-alpine
FROM postgres:17.2-alpine3.21

# BEST (for production, fully reproducible)
FROM node:22.10.0-alpine3.21@sha256:abc123def456...
```

How to find SHA:
```bash
docker pull node:22.10.0-alpine3.21
docker inspect --format='{{index .RepoDigests 0}}' node:22.10.0-alpine3.21
```

## Rule 4: `.dockerignore` is mandatory

Every Dockerfile needs a `.dockerignore` next to it. Otherwise:
- `node_modules` from host bloats build context (huge slowdown)
- `.git` folder leaks repo metadata + history into image
- Secrets in `.env` files end up in images
- IDE files (`.idea/`, `.vscode/`, `*.swp`) bloat

## Rule 5: Layer ordering for cache hits

Each Dockerfile instruction creates a layer. Cache invalidates from any changed instruction onward. So **order from least-changing to most-changing**:

```dockerfile
# Correct order:
FROM node:22.10.0-alpine             # 1. Base (rarely changes)
RUN apk add --no-cache git curl       # 2. System deps (rarely)
WORKDIR /app                          # 3. Directory setup
COPY package.json package-lock.json ./  # 4. Manifest (rarely vs code)
RUN npm ci                            # 5. Install deps (cached if manifest unchanged)
COPY . .                              # 6. App code (changes often)
RUN npm run build                     # 7. Build
CMD ["node", "dist/server.js"]        # 8. Run
```

Common mistake: `COPY . .` BEFORE the `RUN npm ci` step. This invalidates the dep install cache on every code change.

## Rule 6: Layer minimization (but don't go crazy)

```dockerfile
# WRONG — 3 layers, each kept in history with intermediate state
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*

# RIGHT — single layer, clean state preserved
RUN apt-get update && apt-get install -y --no-install-recommends \
    curl \
    && rm -rf /var/lib/apt/lists/*
```

But don't over-minify. Splitting unrelated things into separate RUNs is fine — it helps debugging and parallel BuildKit can sometimes leverage it.

## Rule 7: Healthchecks (for production)

```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --start-period=30s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1
```

Alternatives for images without curl:
```dockerfile
# wget (alpine)
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD wget -qO- http://localhost:3000/health > /dev/null 2>&1 || exit 1

# nc (busybox)
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD nc -z localhost 3000 || exit 1

# Node's built-in (for Node apps without curl/wget)
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD node -e "require('http').get('http://localhost:3000/health', r => process.exit(r.statusCode === 200 ? 0 : 1)).on('error', () => process.exit(1))"
```

## Rule 8: Use `EXPOSE` for documentation

```dockerfile
EXPOSE 3000
```

`EXPOSE` doesn't actually publish ports (that's `-p` at run-time). But it documents intent and is used by `docker run -P` and various orchestrators.

## Rule 9: Labels for traceability

```dockerfile
LABEL org.opencontainers.image.title="My App"
LABEL org.opencontainers.image.description="Description"
LABEL org.opencontainers.image.version="1.2.3"
LABEL org.opencontainers.image.source="https://github.com/user/repo"
LABEL org.opencontainers.image.licenses="MIT"
LABEL org.opencontainers.image.created="2026-06-08T14:00:00Z"
LABEL org.opencontainers.image.authors="Jorge <jerbco@gmail.com>"
```

These show up in `docker inspect` and image registry UIs. Useful for tracking what's running in prod.

## Rule 10: Secrets handling

```dockerfile
# WRONG — secret baked into image, visible in `docker history`
RUN curl -H "Authorization: Bearer SECRET_TOKEN" https://api.example.com/data > /app/data

# RIGHT — use BuildKit's --secret
# In Dockerfile:
RUN --mount=type=secret,id=apitoken \
    curl -H "Authorization: Bearer $(cat /run/secrets/apitoken)" https://api.example.com/data > /app/data

# Build:
DOCKER_BUILDKIT=1 docker build --secret id=apitoken,src=$HOME/.api_token -t myapp .
```

For runtime secrets, use env vars passed at `docker run -e VAR=value` or `-e VAR` (uses host env value).

## Rule 11: Set ENV vars correctly

```dockerfile
ENV NODE_ENV=production
ENV PYTHONUNBUFFERED=1
ENV PATH="/app/bin:${PATH}"
```

Common per stack:
- Node: `NODE_ENV=production`, `NPM_CONFIG_LOGLEVEL=warn`
- Python: `PYTHONDONTWRITEBYTECODE=1`, `PYTHONUNBUFFERED=1`, `PIP_NO_CACHE_DIR=1`
- Go: `CGO_ENABLED=0` (for static binaries, if pure Go)
- JVM: `JAVA_TOOL_OPTIONS="-XX:+UseContainerSupport -XX:MaxRAMPercentage=75.0"`

## Rule 12: WORKDIR over `RUN cd ...`

```dockerfile
# WRONG (cd doesn't persist between RUNs)
RUN cd /app && npm install

# RIGHT
WORKDIR /app
RUN npm install
```

## Rule 13: Handle signals properly (PID 1)

Containers run your CMD as PID 1. PID 1 has special signal handling — it doesn't get default SIGTERM unless it explicitly handles it. This makes `docker stop` hang for 10s, then SIGKILL.

```dockerfile
# Option A: use --init flag at runtime (uses tini as PID 1)
# docker run --init myapp

# Option B: bake tini into image
RUN apk add --no-cache tini
ENTRYPOINT ["/sbin/tini", "--"]
CMD ["node", "server.js"]

# Option C: use a process manager that handles signals (PM2, dumb-init)
```

For Node specifically, `node` does handle SIGTERM since Node 16+, so option A is usually enough.

## Rule 14: Distroless for security-critical workloads

`gcr.io/distroless/*` images have NO shell, NO package manager, NO unnecessary binaries. Minimal attack surface.

```dockerfile
FROM golang:1.23 AS builder
WORKDIR /app
COPY . .
RUN CGO_ENABLED=0 go build -o /app/myapp

FROM gcr.io/distroless/static-debian12
COPY --from=builder /app/myapp /
USER nonroot
CMD ["/myapp"]
```

Pros: ~5 MB image, drastically reduced CVEs.
Cons: no `docker exec sh` to debug — debugging requires sidecar or pre-baking shell into separate debug stage.

## Rule 15: Build args vs env

- `ARG` — available during build only, not in final image
- `ENV` — available during build AND in final image (at runtime)

```dockerfile
ARG NODE_VERSION=22                  # build-time
FROM node:${NODE_VERSION}-alpine
ARG GIT_SHA=unknown                  # passed at build time: --build-arg GIT_SHA=$(git rev-parse HEAD)
ENV APP_GIT_SHA=${GIT_SHA}           # promoted to ENV so runtime can read it
```

## Common patterns to avoid

```dockerfile
# DON'T — these are anti-patterns

ADD https://example.com/file.tar.gz /app/   # use COPY + RUN curl (more explicit, cacheable)

RUN cd /app && npm install                   # WORKDIR /app + RUN npm install

USER root                                    # if you must, switch back: USER node

EXPOSE 80 443                                # privileged ports — use 8080/8443 + map at runtime

COPY . .                                     # then later COPY package.json — should be reverse order

ENV NODE_VERSION 22                          # legacy syntax — prefer ENV NODE_VERSION=22

LABEL maintainer="..."                       # deprecated — use org.opencontainers.image.authors
```
